# 营销网站生成器

## 核心理念

AI 建站工具可在 30 秒到 15 分钟内生成网站，但数据显示 AI 生成的网站仅 **0.026%** 长期保持活跃。问题不在技术，在于**缺乏差异化和场景深度**。

**赢家公式 = 深度定位 + 场景专属设计 + 风格多样性**

---

## 触发后第一步：判断模式

触发后，**立即问这一个问题**，不要直接进入 Step 1：

> 你是有现成资料/明确需求（快速模式），还是需要从头梳理定位（深度模式）？
>
> **A. 快速模式** — 你已经知道自己是做什么的，有基本资料或能用几句话说清楚。我会直接帮你生成，3轮完成。
> **B. 标准模式** — 你有现成资料（文档/图片/链接），但想要有设计文档确认、质量更有保障。5轮左右完成。
> **C. 深度模式** — 你想系统梳理定位、品牌故事、竞争优势，确保网站有差异化。会走完整流程，质量更高但需要更多轮对话。

### 三种模式路径

**A. 快速模式**：6问收集 → Step 6（简化设计文档）⏸️ → Step 7 生成 → Step 8 交付
**B. 标准模式**：基本信息+素材收集 → Step 6（完整设计文档）⏸️ → Step 7 → Step 8
**C. 深度模式**：Step 1 → Step 2 ⏸️ → Step 3 → Step 4 ⏸️ → Step 5 → Step 6 ⏸️ → Step 7 → Step 8

⏸️ = 必须等待用户明确确认，不得自动进入下一步

---

### 沟通规范（全模式适用）

**内部框架代码不暴露给用户**：A1/A2/A3、B1/B2/B3/B4、C1-C5、S1-S27 等编号是 AI 内部决策用的分类标签，**绝不直接写进给用户看的内容**。跟用户说"企业专业风格"，不说"S2"；说"你的客户是 G 端政府采购"，不说"B1"。

**设计文档里的分析框架词禁止出现在网站页面上**：「明线」「暗线」「明线价值」「暗线追求」「深层追求」「核心判断」「定位约束」「三维坐标」等，是设计文档内部的分析标签，**绝对不能作为 section 标题、卡片标题或任何可见文字出现在生成的网站里**。这些内容在转化为网站文案时，必须改写为用户和访客能理解的正常语言，例如「为什么选我们」「我们的承诺」「你将获得什么」。

**每个设计决策都说清楚理由**：用户看到的任何设计判断，都要附一句为什么——不是"我选了深蓝配色"，而是"因为你的主要客户是政府采购和专业运动队，需要信任感和专业度，所以选深蓝而不是活力橙"。

**用用户的语言**：避免设计行业术语（"信息密度"、"七支柱"、"定位约束矩阵"等）直接出现在给用户看的内容里。可以说"页面内容排得稍密一点，信息更充足"，而不是"信息型密度"。

**确认节点要让用户真的能做判断**：给用户看的内容要让他能回答"这个对/不对"，而不是"看不懂但好像挺专业"。每个关键决策都要给用户一个具体的反馈抓手（例如"如果你觉得这个定位方向有偏差，告诉我哪里不对"）。

## Step 2：明暗线探索（必做）

三维坐标确认后，**必须问以下问题**再进入 Step 3。明暗线是全站设计的灵魂，决定了网站不只是"好看的展示页"，而是真正能打动人的品牌表达。

### 什么是明线和暗线？

- **明线**：客户使用你的产品/服务后，能**显性感知到的价值和效果**。是客户一看就懂、一听就心动的表层价值主张。例如：降本增效、管理透明、风险可控、效率提升。明线回答的是"客户能得到什么看得见的好处"。
- **暗线**：你做这件事**更深层的战略意图或内心追求**，不会直接写在广告语里，但会渗透在品牌故事、文案调性和整体叙事中。例如：改变一个行业、构建产业生态、陪伴客户成长。暗线回答的是"你真正想做成什么"。

明线决定首屏价值主张、核心卖点文案和CTA引导语（让客户觉得"这就是我要的"）；暗线决定品牌故事、文案调性和情感共鸣点（让客户觉得"这家公司懂我"）。一个只有明线的网站是冷冰冰的功能介绍页，一个有暗线的网站才能让人记住你。

> **注意**：明线不等于转化动作（加微信/填表单/下单），转化动作是明线的**执行层**，属于CTA设计范畴，在 Step 6 转化设计中处理。明线本身是更上层的**价值主张**。

### 问法（带懒人选项）

```
在开始设计之前，我需要了解两个核心问题：

**Q1 明线（显性价值）**：
客户用了你的产品/服务后，最直观能获得什么效果？
  A. 降本增效（省钱、省时间、减少浪费）
  B. 管理透明（进度可视、数据实时、风险可控）
  C. 品质提升（作品更好、服务更专业、体验更佳）
  D. 增长获客（更多客户、更高转化、品牌提升）
  E. 安心省心（合规、安全、有人兜底）
  F. 你帮我提炼（我说一下业务你来判断）

**Q2 暗线（深层意图）**：
抛开表面的产品功能，你做这件事更深的意图或追求是什么？
  A. 想改变一个行业的运作方式
  B. 想构建一个生态/平台，连接更多人
  C. 想陪伴客户成长，成为长期伙伴
  D. 对品质/美/真实有执着的追求
  E. 想证明一种新的可能性
  F. 没想过这个 / 你帮我聊聊
```

### "帮我选"处理规则

**Q1 帮我提炼**：根据行业 + 产品形态 + 客户痛点，自动提炼客户能显性感知到的核心价值，给出建议并说明理由。

**Q2 帮我聊聊 / 没想过**：
1. 从用户已提供的信息中提炼暗线线索（为什么选这个行业？为什么坚持这么久？你的愿景是什么？）
2. 给出2-3个可能的暗线方向供选择
3. 如果确实找不到暗线，**不要硬编**，标注"暗线待挖掘"，网站先以明线驱动设计，后续可补充

### "暗线待挖掘"判定标准

满足以下任一条件，标注"暗线待挖掘"：
- 用户选了 Q2-F，且追问 2 轮后（"你为什么做这行"、"你最想改变什么"）仍无具体共鸣点
- 用户明确表示"没有，就是做生意"或类似表达
- 从素材中无法提炼出超出功能层面的意图

**标注方式**：在明暗线记录中写 `暗线：待挖掘（明线全面驱动）`

### "暗线待挖掘"时的 Step 6 处理规则

设计文档和 Step 7 生成时：
- 明暗线触点映射表"暗线列"全部填 `—`（不强行填入）
- 品牌故事/About 页改为**行业愿景型**叙事（如"XX 行业正在经历 YY 变化，我们在这里帮助..."）
- Footer 金句改为**行业前景句**（如"XX 行业的下一个十年，从清晰的定位开始"），不填暗线情感句
- 文案整体调性偏功能性和结果导向，少用情感化词汇

### 桥梁词提炼

基于明暗线，尝试找到一个品牌名或Slogan中能同时承载两层含义的"桥梁词"：
- 伙伴云的"伙伴"（明线：合伙人 / 暗线：搭子）
- 蒋苏的"同行"（明线：同行业 / 暗线：一起走）
- 微光里的"微光"（明线：小而美 / 暗线：在黑暗中的一点光）

如果品牌名天然有双关义，直接用；如果没有，在Slogan中体现即可。不强求。

### 输出格式

确认后记录为：
```
明线：[显性价值] — [客户能直观感知的效果]
暗线：[深层意图] — [你真正想做成什么]
转化动作：[CTA类型] — [具体动作，如预约演示/加微信/填表单]
桥梁词：[词] — [明线义/暗线义]（如有）
```

**必须等用户确认明暗线后，再进入 Step 3。**

## Step 3：信息收集（深度模式 C）

### 判断工作模式

- **方式 A（问答模式）**：用户没有现有资料 → 进入下方问答
- **方式 B（资料模式）**：用户提供了文档/素材 → 进入下方素材解读
- **方式 C（问答+资料结合）**：先问答，再读素材补充，合并进入 Step 4

### 方式 A：九大定位问答

```
请回答以下9个问题（任何一题不确定都可以说"你帮我想"，我会根据行业经验帮你推断）：

1. 你是谁？用一句话定义你的业务（不确定→你帮我提炼）
2. 你的客户是谁？（行业、角色、特征）
3. 客户在找你之前，面临什么痛点或困境？（不确定→你帮我分析）
4. 你是怎么帮客户解决问题的？
5. 你的方案有哪些亮点或与众不同的地方？（不确定→你帮我挖掘）
6. 怎么定价？
7. 客户一般怎么评价你？（没有→你帮我基于案例推演）
8. 为什么客户应该选你而不是竞争对手？（不确定→你帮我分析）
9. 客户最常问哪些问题？（没有→你帮我推断常见FAQ）
```

### 方式 B：素材解读

**先确认素材清单**：

```
在查看资料之前，确认一下素材：

1. 有哪些资料？（文件路径或位置）
2. 公司/个人介绍文档？
3. 网页链接？（官网、社媒等）
4. 产品/服务宣传资料？
5. 客户评价、案例记录？
6. 图片素材：
   我会先用 Unsplash 免费图库帮你找配图，让你先看到整体效果。每张图会标注替换标准，你后续可以用自己的照片替换。
   A. 先用占位图，后续再换真实照片（推荐，最快出效果）
   B. 我有现成图片素材：产品实拍照 / 团队或主理人照片 / 工作场景照 / Logo 源文件 / 客户Logo（用于Logo墙），放在 [文件夹路径]
   C. 我有一些图但不确定能不能用，你帮我看看

素材越全，页面越贴合真实业务。
```

确认后逐一读取，提取：主体信息、产品/服务、客户群体、竞争优势、定价、客户评价。

**图片素材处理**：选 A → Step 7 时用 Unsplash 搜索下载占位图；选 B → 用 `ls` 列出文件夹分配图片用途（不逐张 Read，省 token）；选 C → Read 图片评估是否满足使用标准（分辨率 ≥ 1200px 宽、构图合适、与品牌调性一致），不满足的给出具体建议并用 Unsplash 替代。

### 图片素材策略（全模式适用）

以下规则适用于 A/B/C 三种模式的图片处理。**根据产品类型选择不同的视觉策略**。

#### 判断视觉策略类型

| 产品类型 | 视觉策略 | 典型场景 |
|---------|---------|---------|
| **实体产品/线下服务** | 摄影驱动型 | 餐饮、健身、教育培训、房产家装、零售、制造业 |
| **软件/SaaS/App** | 产品展示型 | 管理系统、效率工具、开发者平台、数据产品 |
| **混合型** | 摄影+产品结合 | 有实体场景的科技公司、有线下门店的 SaaS |

---

#### 策略一：摄影驱动型（实体产品/线下服务）

**收集优先级**：Unsplash 占位（推荐，最快出效果）→ 用户真实照片替换 → 品牌VI素材补充

**Unsplash 搜索执行方式**（不需要 API key，不需要写脚本）：
1. 用 WebSearch 搜索 `site:unsplash.com {行业关键词} {场景关键词} {情绪关键词}`（英文关键词效果更好）
2. 用 WebFetch 获取 Unsplash 页面，提取图片原始 URL
3. 用 `curl -L -o assets/{page}_{element}_{desc}.jpg "{unsplash_url}?w=1600&fit=crop"` 下载到 assets/

**搜索关键词组合示例**：
- 咖啡品牌 → `coffee shop interior warm lighting`
- 教育培训 → `classroom learning study`
- BMX 竞速 → `bmx racing track cycling`
- 健身品牌 → `gym fitness training workout`

**每种 Section 的视觉要求**：

| Section 类型 | 视觉元素 | 说明 |
|-------------|---------|------|
| Hero（首屏） | 1 张全幅背景图 | 必须有视觉冲击力，禁止纯色背景 |
| 关于/品牌故事 | 1-2 张照片 | 主理人照/团队照/工作场景 |
| 产品/服务展示 | 每项 1 张照片 | 产品实拍或使用场景图 |
| 案例/客户证言 | 每个案例 1 张 | 客户照片、项目截图或成果图 |
| 照片分隔带 | 1 张全幅氛围图 | 与品牌调性一致，可做视差滚动 |
| Logo 墙 | 6+ 个 Logo | 灰度统一处理 |

---

#### 策略二：产品展示型（软件/SaaS/App）

**软件产品不应该用 Unsplash 实拍照填满页面。** 视觉重心是产品本身的界面、功能流程和使用场景，而非氛围照片。

**视觉元素优先级**（按推荐程度排序）：

**① 内联 SVG 插画**（最推荐，零外部依赖）
- 直接在 HTML `<body>` 中绘制 SVG 插画：图标、流程图、功能示意图、抽象装饰
- 颜色使用 CSS 变量（`var(--color-primary)`），自动跟随品牌配色
- 适合：功能介绍区、工作流程区、优势对比区
- 风格参考：线条图标风、几何抽象风、等距风（isometric）

**② CSS 设备框 + 产品截图占位**
- 用纯 CSS 绘制浏览器窗口框/手机框/平板框，内嵌产品界面截图或占位色块
- Hero 区推荐：左侧文案 + 右侧产品截图在浏览器框中
- 用户后续替换真实产品截图即可
- CSS 设备框模板（在 HTML 中直接实现）：
  ```html
  <!-- 浏览器窗口框 -->
  <div class="browser-mockup">
    <div class="browser-header">
      <span class="browser-dot"></span>
      <span class="browser-dot"></span>
      <span class="browser-dot"></span>
    </div>
    <div class="browser-content">
      <img src="assets/product_screenshot.png" alt="产品界面" loading="lazy" />
    </div>
  </div>
  ```

**③ CSS/SVG 渐变与几何背景**
- Hero 区用渐变网格（gradient mesh）或几何图案代替全幅照片
- 中间 section 用带透明度的色块、噪点纹理、几何装饰
- 可用 CSS `radial-gradient`/`conic-gradient` 组合生成，不需要外部图片

**④ unDraw 开源插画**（需要下载 SVG 文件）
- 来源：https://undraw.co —— 免费商用，无需署名
- 颜色可在网站上自定义后下载，匹配品牌主色
- 搜索执行：WebSearch `site:undraw.co {关键词}` → WebFetch 页面 → 下载 SVG 到 assets/
- 适合：关于页、功能介绍、空状态、404 页面

**每种 Section 的视觉要求（软件产品版）**：

| Section 类型 | 视觉元素 | 说明 |
|-------------|---------|------|
| Hero（首屏） | 左文案 + 右产品截图（CSS设备框）或渐变背景 + 居中文案 | 禁止全幅实拍照；用渐变/几何/产品截图 |
| 功能/特性展示 | 每项 1 个内联 SVG 图标或小插画 + 文案 | 左文右图或图文交替排列 |
| 工作流程 | 步骤编号 + SVG 图标 + 连接线 | 纯 CSS/SVG 实现，不需要照片 |
| 产品截图展示 | CSS 设备框 + 截图占位 | 用户后续替换真实截图 |
| 数据/成就区 | 数字 + 内联 SVG 图标 | 背景可用渐变或深色 |
| 客户证言 | 头像占位圆 + 引用文字 | 小头像即可，不需要全幅照片 |
| 定价/对比 | 纯 CSS 卡片 | 不需要图片 |

**布局要求**（软件产品特有）：
- **禁止全部 section 都用全幅大图** —— 至少 50% 的 section 使用「左文右图」或「图文交替」布局
- Hero 区推荐使用不等分布局（如 `grid-template-columns: 1.2fr 1fr`），文案占大边
- 功能展示区推荐「左文右小图」交替排列（奇数行左文右图，偶数行左图右文）
- 产品截图不做全幅出血，放在设备框中有呼吸感

---

#### 通用规则（两种策略都适用）

**人物照片约束**：
- **除非内容明确面向海外/国际受众，人物照片必须使用亚洲面孔**
- Unsplash 搜索时加 `asian` 关键词（如 `asian business team meeting`）
- 禁止典型 stock photo 感：白背景摆拍、过度职业微笑、西装握手
- 搜索优先级：真实工作场景 > 生活方式 > 肖像特写

**用户自有图片评估标准**（选 C 时适用）：
- 分辨率：宽 ≥ 1200px（Hero 背景图 ≥ 1600px）
- 构图：主体清晰，无严重偏色/模糊/过曝
- 调性：与品牌定位和视觉风格一致
- 不符合标准的 → 给出具体改进建议，并用 Unsplash 或 SVG 插画替代

## Step 6：生成设计文档

### 6.1 反趋同检查

查阅 [references/case-library.md](references/case-library.md)，找最接近的历史案例，写出**创意声明**——至少 3 处差异。

### 6.2 五层设计决策

参考 [references/design-guide.md](references/design-guide.md)：

1. **定位约束矩阵**（A×B×C → 组合）→ "必须做什么 / 必须避免什么"
2. **视觉风格选择**（用户偏好 + 行业匹配 + 定位约束）→ 主风格 + 辅风格
3. **页面积木库** → 页面数量和组合（单页 or 多页）
4. **Section 布局模式库** → 每页内部节奏（相邻 section 不重复）
5. **七支柱设计规则** → 转化、信任、情感、信息架构全面达标

### 6.3 七支柱融入设计

| 支柱 | 设计文档中必须回答 |
|------|-----------------|
| **视觉层级** | 首屏 50ms 传达什么？3 个对比层级是什么？ |
| **内容策略** | 每页单一目标是什么？标题叙事性还是描述性？ |
| **信任可视化** | 信任证据占比多少？用什么形式？ |
| **转化优化** | CTA 在首屏什么位置？文案用什么人称？ |
| **情感设计** | 色彩触发什么感受？有什么微交互？ |
| **信息架构** | 导航用用户语言？信息密度选哪种？ |
| **可访问性** | 对比度达标？语义 HTML？ |

### 6.4 反AI感设计规则（每个项目必须遵守）

**字体**：
- 禁止使用 Inter、Roboto、Arial、system-ui 作为主字体（按 technical-spec.md 字体表严格匹配风格）
- 标题字体与正文字体必须形成明确对比（粗细、字体家族、大小至少差 2 级）
- 连续两个项目不得使用相同字体组合

**色彩**：
- 禁止"安全配色"：浅灰底+蓝色按钮、白底+紫色渐变、全站只有黑白灰
- 主色必须大胆占主导地位，辅色用于锐利点缀（accent），不均匀分配
- 设计文档创意声明中写明"为什么不用这个行业最常见的配色"
- 背景 ≠ 白色或 #f5f5f5——可以用深色、带色调的浅色、渐变、微纹理

**空间构成**：
- 至少 1 个 section 使用非对称布局（文字偏左/右 + 图片不等分、对角线流动）
- 至少 1 个元素打破网格（跨列、图片出血到边缘、元素重叠）
- section 间距不得全部相同——至少 3 种不同的垂直节奏

**背景与氛围**：
- Hero 区禁止纯色背景——必须有以下至少一种：全幅背景图 + overlay、渐变网格、噪点纹理、几何图案
- 全站至少 2 个 section 使用非纯色背景（渐变、半透明层叠、纹理、背景图）
- 可选装饰元素：grain overlay、SVG 几何装饰、品牌纹样

**动效**：
- 首屏入场必须 staggered reveal：标题 → 0.2s 副标题 → 0.4s CTA → 0.6s 装饰元素（不是所有元素同时 fade-in）
- 按钮 hover 必须有 transform + box-shadow 组合变化
- 卡片 hover 必须有 translateY(-4px) + shadow 加深

**反趋同声明**（写入设计文档创意声明）：
- 写出"这个设计不像典型AI生成网站的 3 个理由"
- 每个理由对应一个具体的、可在代码中验证的设计决策

### 6.5 亚洲设计适配

中国市场建议：标准型或信息型密度，多CTA，信息适度前置。

[每页一行：页面名 → 核心目标]

## Step 7：生成页面

技术架构规范参考 [references/technical-spec.md](references/technical-spec.md)。

### 生成前准备

**⚠️ 生成文件必须放到 skill 目录之外。**
- 所有项目文件（HTML、CSS、JS、assets/ 等）生成到用户指定目录，或默认放到 `~/Desktop/[品牌名]-website/`
- **禁止在 skill 目录内（`hb-website-creator/`）创建任何项目文件**，包括临时文件

### 生成约束（每页必须遵守）

1. **色彩**：严格按设计文档色彩方案，所有颜色通过 CSS 变量，禁止硬编码
2. **文案**：直接来自商业定位文档，不自由发挥或编造数据
3. **结构**：Section 顺序严格按设计文档规划
4. **首屏**：必须包含「标题 + 副标题 + CTA 按钮」三要素
5. **CTA**：每页至少 2 个，文案用第一人称行动导向（"立即咨询" > "联系我们"）
6. **导航栏**：Navbar 必须包含「登入/注册」按钮（`<a href="#">`，链接后期替换），与主CTA按钮并列
7. **自检**：每个页面完成后，列出该页的 CTA 数量和位置，确认 ≥ 2 个；确认 Navbar 包含登入/注册按钮
8. **视觉素材使用**（根据图片素材策略中判定的类型执行）：
   - **摄影驱动型**（实体产品/线下服务）：
     - Hero section 必须使用全幅背景图（CSS `background-image` + 半透明遮罩层确保文字可读）
     - 每页至少 1 个 section 包含照片元素
     - 如设计文档有照片分隔带，使用 `background-attachment: fixed` 视差效果
     - Unsplash 占位图在生成阶段下载：WebSearch → WebFetch → `curl` 下载到 `assets/`
   - **产品展示型**（软件/SaaS/App）：
     - Hero section 使用渐变/几何背景 + 产品截图在 CSS 设备框中（或居中文案 + 渐变背景），**禁止全幅实拍照**
     - 功能展示区使用内联 SVG 插画或图标，采用「左文右图」交替布局
     - CSS 设备框（浏览器窗口/手机框）在 HTML 中直接实现，内嵌产品截图占位
     - 可用 unDraw (undraw.co) 下载 SVG 插画到 assets/
     - **禁止全部 section 都用全幅大图** —— 至少 50% 使用图文并排布局
   - **通用**：图片用 `<img src="assets/...">` 或 CSS `background-image: url(assets/...)` 引用
9. **反AI感实施**：
   - 首屏入场动画使用 staggered `animation-delay`（标题 0s → 副标题 0.2s → CTA 0.4s → 装饰元素 0.6s）
   - 至少 1 个 section 使用 CSS Grid 不等分列（如 `grid-template-columns: 1fr 1.5fr`）
   - 背景处理：Hero 用 `linear-gradient` overlay + `background-image`；至少 1 个中间 section 用渐变或纹理背景
   - 按钮 hover 必须有 `transform` + `box-shadow` 组合效果；卡片 hover 必须有 `translateY(-4px)` + shadow 加深

### 生成顺序

逐页生成，格式：
```
✅ [页面名].html 完成 — CTA 位置：[首屏按钮] + [底部按钮]
进度：X/X 页
```

全部完成后进入 Step 8。

## Step 8：交付检查

所有页面生成后，**必须执行以下 4 个动作**（不是自我对照清单，是真实操作）：

### 必须做的 4 件事

**① CTA 可见性检查**
打开 `index.html` 代码，找到 Hero section，确认 CTA 按钮在首屏 fold 以上。如果不在，立即移动位置。

**② 色彩一致性检查**
在所有 HTML 文件中搜索硬编码颜色（如 `#FF0000`、`color: red`），发现即修复，改为 CSS 变量。

**③ 导航完整性检查**
确认每个 HTML 文件的导航栏链接都指向正确文件，不存在死链或路径错误。确认每页 Navbar 都包含「登入/注册」按钮。

**④ 图片完整性检查**
- 确认 `assets/` 目录包含设计文档图片策略中规划的所有图片文件
- 确认 Hero section 使用了背景图（不是纯色背景）
- 确认每个页面至少有 1 个 section 包含图片元素
- 确认 Unsplash 图片已下载到 `assets/` 并按 `{page}_{element}_{desc}.{ext}` 正确命名

发现问题 → 立即修复 → 修复后标注 `[已修复]`。
