# 1.5 增加设计规范和 demo 预览策略

**状态**：已完成 · 2026-04-14

## 问题来源

通过多次实际生成观察到的两个核心问题：

**问题一：大字报**。生成的网站标题字号失控（Hero 标题常见 5rem+），中文大字铺满屏幕，视觉上像政治宣传海报而非专业营销站点。根本原因是 skill 只定义了字体家族（用什么字体），但没有定义任何尺寸约束——字号、行高、容器最大宽度、间距全靠 Claude 自由发挥。

**问题二：布局对齐失控**。没有统一间距系统，section padding、元素 gap、margin 随意取值（如 `padding: 7.3rem 0`），导致不同 section 之间节奏不统一，整体视觉凌乱。

**隐性问题**：用户在 Step 4 确认设计文档（纯文字描述）后直接进入全量生成，完整生成一个网站需要大量时间和 token，用户只有做完才能看到视觉效果。如果效果不满意，所有工作全部推倒重来，成本极高。

---

## 本次迭代核心思路

**两个并行改动，分别解决两个问题。**

**改动一（解决大字报）**：在 `references/technical-spec.md` 增加完整的排版与间距规范，用 CSS 变量 + `clamp()` 函数锁定字号上限，定义容器最大宽度和 8px 基准间距 scale，并在 Step 5 生成约束中强制引用。

**改动二（解决效果确认问题）**：在 Step 4（设计文档确认）和 Step 5（全量生成）之间插入 Step 4.5 首屏打样。打样生成 3 个视觉方向版本的 demo（首屏 Hero + 次屏内容 Section + 底部全套 UI 样式表），用 Tab 切换器展示，用户选定方向后再进入正式生成。

**设计原则**：3 版 demo 共享同一份设计文档（配色、字体、文案），差异只在视觉处理层面（字重、间距、色彩运用方式、背景处理），让非设计师用户能通过直觉快速做判断。

---

## 迭代内容

### 1. 排版与间距规范（新增至 `technical-spec.md`）

在 CSS 变量模板区域（`.site-widget {}` 块内）增加三组变量：

**字号层级**（`clamp()` 响应式，上限锁定）：
```less
--text-hero:    clamp(2rem, 5vw, 3.25rem);          /* Hero 主标题，max 52px */
--text-h1:      clamp(1.75rem, 4vw, 2.5rem);         /* 页面标题，max 40px */
--text-h2:      clamp(1.5rem, 3vw, 2rem);             /* Section 标题，max 32px */
--text-h3:      clamp(1.125rem, 2vw, 1.5rem);         /* 子标题，max 24px */
--text-body:    clamp(0.9375rem, 1.2vw, 1.0625rem);   /* 正文 15-17px */
--text-sm:      clamp(0.8125rem, 1vw, 0.875rem);      /* 辅助文字 13-14px */
--text-xs:      0.75rem;                               /* 标签/注释 12px */
```

**行高**：
```less
--leading-tight:   1.2;    /* 标题 */
--leading-snug:    1.4;    /* 副标题/短文案 */
--leading-normal:  1.6;    /* 中文正文 */
--leading-relaxed: 1.8;    /* 长段落 */
```

**容器最大宽度**（解决标题铺满全屏）：
```less
--max-w-page:    1200px;   /* 整站内容区 */
--max-w-text:    720px;    /* 纯文字段落 */
--max-w-hero:    800px;    /* Hero 文字块 */
```

**间距 scale**（8px 基准）：
```less
--space-1: 0.5rem;  --space-2: 1rem;   --space-3: 1.5rem;  --space-4: 2rem;
--space-6: 3rem;    --space-8: 4rem;   --space-12: 6rem;   --space-16: 8rem;
```

**使用规则**：
- Hero 标题必须用 `--text-hero`，h1 用 `--text-h1`，以此类推——禁止硬编码 `font-size` 超过 `1rem`
- 内容容器必须 `max-width: var(--max-w-page); margin: 0 auto`
- Hero 文字块必须 `max-width: var(--max-w-hero)`
- Section `padding` 必须从间距 scale 中选取，禁止任意数值

### 2. Step 4 设计文档工作版新增「排版微调」字段

允许在默认 scale 基础上做风格化调整，但必须显式声明，例如：
```
## 排版微调
- Hero 字号：使用默认 --text-hero（无调整）
- Section padding：标准页用 --space-12，首屏用 --space-16
```

### 3. Step 4.5 首屏打样（全新步骤）

**流程定位**：

| 模式 | Step 4.5 |
|------|----------|
| 快速模式 A | 跳过（除非用户主动要求） |
| 标准模式 B | 推荐，询问用户是否要看 |
| 深度模式 C | 必做 |

**每版 Demo 包含 3 块**：
1. 首屏 Hero（Navbar + 标题 + 副标题 + CTA + 背景处理）
2. 次屏内容 Section（3-4 个特性/价值卡片）
3. 底部全套 UI 样式表（色板实际色块、字体预览、按钮样式含 hover、卡片样式、间距节奏可视化）

**三版差异轴**（同一套配色/字体/文案，视觉处理不同）：

| 版本 | 名称 | 字体表现 | 空间节奏 | 色彩运用 | 背景处理 |
|------|------|---------|---------|---------|---------|
| A | 沉稳大气 | 字重 700-900，字号靠近 clamp 上限 | 宽松留白，间距大 | 克制——主色仅用于 CTA | 深色/暗调 overlay |
| B | 轻盈明快 | 字重 500-600，字号居中 | 紧凑适中 | 活泼——辅色出现在背景色块 | 浅色/带色调柔和背景 |
| C | 冲击力强 | 字重 900，标题正文对比悬殊 | 非对称，打破网格 | 大胆——主色大面积使用 | 全幅图/大胆渐变 |

**技术实现**：
- 在项目目录生成 `widget.jsx` + `style.less` + `main.jsx`（同技术栈）
- 顶部固定 Tab 切换器（A/B/C），点击切换时整页被 `.variant-a/b/c` 包裹
- 三个变体通过 CSS 变量覆盖实现差异，核心内容结构相同
- 执行 `npm install && npm run dev` 启动本地预览

**用户选择流程**：
```
展示预览 → 用户选 A/B/C → 记录到设计文档工作版 → 进入 Step 5（覆盖打样文件）
            ↓ 都不满意
            问"哪个最接近？想改什么？"→ 生成第 4 版
            最多 2 轮修改（共 5 版），之后要求用户提供参考截图/链接
```

选定后 Step 5 从头覆盖生成正式文件（打样文件不复用，结构不同）。

### 4. Step 5 生成约束新增排版条目

在现有 9 条约束之后增加第 10 条：
```
10. **排版**：严格使用 technical-spec.md 排版变量（--text-hero 等），
    禁止硬编码 font-size 超过 1rem；Hero 文字块必须设置 max-width: var(--max-w-hero)；
    section padding 必须从间距 scale 中取值。
    如 Step 4.5 已执行，项目目录已存在 node_modules，直接覆盖 widget/style/main 文件，不重复 npm install。
```

### 5. 流程状态机与 Gate 表更新

流程图三条路径（A/B/C）加入 Step 4.5 节点，Gate 表增加一行：

| Step 4.5 | 输入：设计文档 | 输出：3 版打样 + 用户选择 | Gate：用户选定版本（⏸️ 深度模式，可选其他模式） |

## Token 影响评估

| 项目 | Token 变化 |
|------|-----------|
| `technical-spec.md` 排版规范新增文本 | +约 800 tokens（常驻上下文） |
| `SKILL.md` Step 4.5 章节新增文本 | +约 2000 tokens（常驻上下文） |
| Step 4.5 打样生成（3 版 widget.jsx + style.less + main.jsx） | +约 8000-12000 tokens（仅执行打样时） |
| npm install + 启动预览服务 | +约 200 tokens |
| 用户修改 1 轮（第 4 版） | +约 4000-6000 tokens |
| **总计（深度模式，含 1 轮修改）** | 固定 +2800；打样时 +12000-18000 |

打样步骤对标准/快速模式是可选的，不强制执行不产生额外成本。深度模式必做，但能避免全站推倒重来（全站生成约 30000-50000 tokens），整体仍是正收益。
